模块拆解

platform-admin

后台模块，也是系统的核心，用来开发后台管理系统。

前端由Vue编写，后端Spring Boot

使用自定义注解+Aop的方式封装日志管理、Redis管理逻辑

自定义注解：

@Target(ElementType.METHOD)

  • 作用：指定注解的适用范围
  • 取值解析：
    • ElementType.METHOD：表示该注解只能标注在方法上
    • 其他常用取值：TYPE（类）、FIELD（字段）、PARAMETER（参数）等

@Retention(RetentionPolicy.RUNTIME)

  • 作用：控制注解的生命周期
  • 策略对比：
    • SOURCE：仅存在于源码阶段（编译器使用）
    • CLASS：保留到class文件（默认值）
    • RUNTIME：运行时可通过反射获取（最常用）

@Documented

  • 作用：标注的注解会包含在Javadoc生成的文档中

日志处理切面类

  • **@Before**：在目标方法执行前执行。
  • **JoinPoint**：封装了目标方法的信息（如方法签名、参数等）。
  • **MethodSignature**：获取方法签名，进一步获取方法名和参数。
  • @SysLog 注解：通过反射获取注解信息，提取操作描述。
  • 日志信息：
    • 注解上的描述：通过反射获取注解及注解里的value值，例如修改密码
    • 方法名：className + "." + methodName + "()"
    • 请求参数：将第一个参数序列化为 JSON 字符串。
    • IP 地址：通过 HttpServletRequest 获取客户端 IP。
    • 用户名：通过 Shiro 工具类获取当前登录用户。
    • 创建时间：当前时间。

Redis切面类

  • 作用：通过 AOP 拦截带有 @RedisCache 注解的方法，实现缓存逻辑。

  • 流程：

    1. 定义切点（@Pointcut）拦截带有 @RedisCache 注解的方法。
    2. 在方法执行前后（@Around）检查缓存是否存在：
      • 如果缓存存在，直接返回缓存数据。
      • 如果缓存不存在，调用目标方法，并将结果存入缓存。
    3. 支持自定义缓存键（cacheKey）和是否固定键（destine）。
    • **@Around**：在目标方法执行前后执行，可以控制目标方法的执行。
    • **ProceedingJoinPoint**：封装了目标方法的信息（如方法签名、参数等）。
    • **MethodSignature**：获取方法签名，进一步获取方法名和参数类型。
    • @RedisCache 注解：通过反射获取注解信息，提取缓存键和是否固定键。
    • 缓存键生成：
      • 如果 destine 为 true，直接使用 cacheKey 作为缓存键。
      • 如果 destine 为 false，根据类名、方法名和参数生成缓存键。
    • 缓存逻辑：
      • 如果缓存不存在，调用目标方法并将结果存入缓存。
      • 如果缓存存在，从缓存中获取数据并反序列化为目标方法的返回类型。

  • 使用J2Cache ，J2Cache 默认使用 Caffeine 作为一级缓存，使用 Redis 作为二级缓存

    • default=1000, 60m,定义缓存名 default ，对象大小 1000，缓存数据有效时间 60 分钟
    • sysCache=1000, 60m,定义缓存名 sysCache，对象大小 1000，缓存数据有效时间 60 分钟
    • shopCache=1000, 60m,定义缓存名 shopCache，对象大小 1000，缓存数据有效时间 60 分钟
    • 使用 J2Cache 的主要原因是通过本地缓存提升性能，分布式缓存保证数据一致性,在电商系统的商品详情页、秒杀活动等，需要快速响应用户请求的高并发场景下本地缓存可以显著减少对数据库和分布式缓存的压力。订单系统、库存系统等，需要保证缓存和数据库的数据一致性，在 数据一致性要求高的场景，J2Cache 的消息通知机制可以及时更新本地缓存。

controller层

使用Shiro进行权限管理

  • 传统 RBAC 三表：用户表（sys_user）、角色表（sys_role）、用户-角色关联表（sys_user_role）。
  • 扩展权限表：sys_menu 表本身承担了权限定义功能，通过 sys_role_menu 表实现角色-权限关联。
  • 命名规范：模块:操作（如 user:delete），与代码中 @RequiresPermissions 注解匹配。
  • 动态解析：通过 sys_menu.perms 字段存储多个权限，逗号分隔。

在用户登录后，通过Shiro获取用户id，根据用户id递归查询用户可以查看的页面列表并返回给前端显示。

同时在各个后端接口上加上@RequiresPermissions注解防止越权访问。

使用一个PageUtil来进行后端分页。

具体流程：角色管理

当创建新的角色时，在 sys_role 表中插入角色信息，并在 sys_role_menu 表中插入角色与菜单权限的关联。创建用户时分配角色，在 sys_user_role 表中插入用户与角色的关联。

  • 用户登录后，系统加载用户的角色和权限信息。在业务逻辑前使用 @RequiresPermissions 注解校验用户权限。如果用户无权限，抛出 UnauthorizedException 并返回 403 状态码，防止越权访问。
  • 用户登录后，查询角色权限并将角色权限缓存到 J2Cache 中，优化权限验证的性能。
  • 当用户访问前端界面时，查询菜单树并将菜单树缓存到 J2Cache中，减少数据库查询。
  • 当角色权限变更时，先对MySQL数据库中的相关字段进行修改，然后在缓存中清除该角色的权限信息和菜单树信息。保证权限变更的实时生效

为什么使用shiro

  • shiro可以通过在方法上加 @RequiresPermissions注解的方式来细分权限管理的粒度
  • shiro是有状态的，更加安全，可以更好的应对用户退出登录、修改密码、修改用户角色等场景

介绍一下shiro的原理

认证
  • 使用username和password组装UsernamePasswordToken，调用login方法
  • login方法会首先使用token去shiro缓存中查询登录的缓存信息，如果没有命中就调用我们自己编写的Realm类中重写的doGetAuthenticationInfo方法
  • 根据token中的用户名和密码去数据库中查询用户，如果命中则将用户放入session中，设为当前用户
  • 查询用户的权限信息，放入J2Cache中
授权
  • 根据用户id查询用户权限信息，如果缓存命中则直接返回，未命中就去数据库中获取
  • 对比用户是否拥有该权限

J2Cache的原理

首次调用 getList 接口时，无法从缓存中获取数据，然后从“数据库”中获取数据库，先是存储到了二级缓存 Redis 中。

第二次调用 getList 接口时，是从二级缓存 Redis 中获取的数据，同时底层会把数据存储到一级缓存 Caffeine 中。

后续调用 getList 接口时，都是直接从一级缓存 Caffeine 中获取数据。

对数据更新后，需要删除缓存中的脏数据，先删除二级缓存，再删除一级缓存

加密算法

使用sha256哈希算法，哈希算法无法还原

  • sha256算法的哈希值长度为256位，提高了暴力破解的难度
  • 将密码通过sha256计算出哈希值后与数据库中存储的密码进行比对

可以改进的点

  • 密码在前端到后端的通信过程中进行加密，可以使用RSA非对称加密，登录前先向服务器发送预热请求，服务器生成完公钥后发送给前端，前端使用公钥加密后再进行传输
