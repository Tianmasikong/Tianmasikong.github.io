Spring

IoC

控制反转是一种设计思想。就是将原本在程序中手动创建对象的控制权交由Spring框架来管理。最直接的体现就是不用去new一个对象了，而是由Spring容器来管理一个叫bean的东西，控制它的整个生命周期。Spring通过Java的反射机制来实现对其他对象的控制，包括初始化、创建、销毁等，解放手动创建对象的过程，同时降低类之间的耦合度。

IoC的好处

资源不由使用资源者管理，而有不使用资源的第三方管理。

  • 资源集中管理，实现资源的可配置和易管理
  • 降低了使用资源双方的依赖程度，也就是耦合度

DI（依赖注入）

IOC最常见的实现方式是依赖注入，指的是在Spring创建对象的过程中，把对象依赖的属性注入到对象中。

在SpringBoot中可以使用@Autowired注解进行依赖注入

也可以使用set的方式，在xml文件中注入

有参构造也是可以的

Bean

Bean代指的就是那些被IoC容器所管理的对象。我们需要告诉IoC容器帮助我们管理哪些对象，这个是通过配置元数据来定义的。配置元数据可以是 XML 文件、注解或者 Java 配置类。

创建Bean的方式

  • 直接在xml文件中声明一个类为Bean
  • 使用@Component,@Service,@Controller,@Repository注解
    • @Component : 侧重于通用的Bean类
    • @Service：标识该类用于业务逻辑
    • @Controller：标识该类为Spring MVC的控制器类
    • @Repository: 对应持久即Dao层，只有属性和Setter,Getter
  • 使用@Bean注解,这种方式用在Spring Boot 应用中
  • 使用注解@Import,也会创建对象并注入容器中

@Component 和 @Bean 的区别是什么？

  • @Component注解作用于类，@Bean注解作用于方法
  • @Component通常是通过类路径扫描来自动侦测以及自动装配到Spring容器中。@Bean注解通常是在标有该注解的方法中定义产生这个bean，告诉Spring这是某个类的实例。
  • @bean注解比@Component注解的自定义性更强，比如引用第三方库中的类时只能用@Bean

注入Bean的注解有哪些？

@Autowired @Resource @Inject

@Autowired和@Resource的区别是什么

  • Autowired属于Spring内置的注解，默认的注入方式为byType，也就是按类型匹配，优先根据接口类型去匹配并注入Bean。但如果出现某个类型下有好几个实现类的情况的话就会出错，这种情况下就要再加个@Qualifier注解来显式指定名称。
  • Resource属于JDK提供的注解，默认注入方式为byName，有两个比较重要的属性:name(名称)，type(类型)。可以通过name属性来显示指定名称。

Bean的作用域有哪些？

通过加个@scope注解来配置bean的作用域

  • singleton：容器中只有唯一的bean实例。默认都是单例的。
  • prototype：每次获取都会创建一个新的 bean 实例。
  • request ：每次HTTP请求都会产生一个新的bean，该bean仅在当前的HTTP request中有效。
  • session：每次来自新的session的HTTP请求都会产生一个新的bean，该bean仅在当前的HTTP session内有效。
  • application/global-session：每个Web应用在启动时创建一个Bean，该Bean仅在当前应用启动时间内有效。
  • websocket：每一次websocket会话产生一个新的bean。

单例 Bean 的线程安全问题了解吗？

单例Bean存在线程问题，主要是因为当多个线程操作同一个对象的时候是存在资源竞争的。

解决方法：

  1. 在Bean中尽量避免定义可变的成员变量。
  2. 在类中定义ThreadLocal成员变量，将需要的可变成员变量保存在ThreadLocal中。

但一般的Bean实际上都是无状态的，如Dao，Service，所以大部分情况下Bean是线程安全的。

AOP

将那些与具体业务实现无关，却被业务共同调用的部分封装起来，减少系统的重复代码，是开闭原则的体现。比如在一些业务中需要在执行之前验证用户权限，验证权限的实现逻辑其实是一样的，就可以把他们抽象成一个切面，然后由Spring统一注入到业务代码中。

AOP有哪些实现方式？

静态代理

基于XML配置的Spring AOP，采用声明的方式实现，在XML文件中配置，大致步骤为：配置文件中配置pointcut，在java中编写实际的aspect类，针对切入点进行相关的业务处理。

通俗来说就是在类中写好方法，然后在xml里配置切点，切面和通知等。

动态代理

主要就是使用使用aspectJ的注解来进行创建，通过加上**@Aspect注解表明这是个切面，在默认构造函数上加上@PointCut表明切入点表达式，然后在那些通知方法上加上@Before**,@Around这种注解声明在什么时候使用这个通知。

这种方式也要在xml文件中声明spring对@AspectJ的支持并声明通知类。

AspectJ 定义的通知类型有哪些？

  • before:目标对象的方法调用之前触发
  • after:目标对象的方法调用之后触发
  • afterReturning:目标对象的方法调用完成，在返回结果值之后触发
  • afterThrowing:目标对象抛出异常后触发
  • around：可以任意的在目标对象的方法调用前后触发

Spring AOP的实现原理

通过动态代理实现。如果我们为Spring的某个bean配置了切面，那么Spring在创建这个bean的时候，实际上创建的是这个bean的一个代理对象，后续对bean中方法的调用，实际上调用的是代理类重写的代理方法，AOP使用了两种动态代理，分别是JDK的动态代理和CGLib的动态代理。

JDK动态代理和CGLIB动态代理的区别？

  1. JDK动态代理使用jdk中的类Proxy来创建代理对象，使用反射技术来实现，不需要导入其他的依赖。CGLIB需要导入相关依赖，使用字节码增强技术来实现。
  2. 当目标类实现了接口的时候，默认使用JDK动态代理方式来增强方法，没有实现接口的时候使用CGLIB动态代理方法来增强方法。
